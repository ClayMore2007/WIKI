# Obsidian + Claude Code 内容创作自动化流程

这条资料的价值不在于“又收了一个 Obsidian 视频”，而在于它把本地知识库、Claude Code、创作规则、Git 版本控制和外部发布工具串成了一个可迭代的内容生产系统。

视频作者对 Karpathy Wiki 的核心补充是：知识库不只用来收集网页链接，还要让 AI 读懂本地项目资料、遵守个人规则、持续吸收反馈，并把这些反馈变成下一轮创作的约束。

## 一句话结论

从 ClayMore 的角度看，这套流程可以抽象为：

`资料沉淀 -> 项目规则 -> AI 执行 -> 人工反馈 -> 规则更新 -> Git 保护 -> 发布/同步`

它更接近一个“项目型知识库”工作流，而不是单纯的剪藏整理流程。

## 流程拆解

### 1. 先建立项目知识边界

视频以“小李飞刀漫画创作”为例，先把原著 PDF / 文本、角色设定、章节拆解、画风要求、人物状态等资料放入本地 Obsidian 项目目录。

关键点不是文件越多越好，而是让每类文件有明确职责：

- 原始资料：保留小说或参考文本。
- 角色档案：记录外貌、服装、武器、性格、色彩等稳定设定。
- 剧情拆解：按章节或场景拆成摘要、人物、地点、关键台词和分镜建议。
- 状态索引：记录人物状态、阶段变化和可点击链接。
- 工作说明：告诉 AI 每次创作前必须读取哪些文件、遵守哪些约束。

对应到 ClayMore，可以复用为“主题项目包”：每个长期主题至少有 raw 来源、主题页、规则/判断标准、索引和更新日志。

### 2. 用反馈训练 AI 的创作标准

作者的第一轮漫画生成并不稳定，出现了人物不一致、风格不一致、台词不准等问题。真正的改进来自每次人工反馈之后，把偏好和错误写回规则文件。

可复用原则：

- 不要只在聊天里纠正 AI，要把高频纠正沉淀进项目规则。
- 喜欢的页面、画风、表达方式要变成 style guide 或标准示例。
- 失败案例要转成负面约束，例如“不改写原文台词”“人物表情必须符合场景”。
- 当参考文件越来越多时，需要一个总入口文件，要求 AI 先读入口，再按任务读取子文件。

这和 ClayMore 的规则页优化方向一致：先读快速入口，再进入具体流程，不让 AI 在一堆规则里迷路。

### 3. 把重复任务封装成 Skill / 工作流

视频里作者进一步把项目工作流放进 Claude Code 的 Skill 或类似可触发流程中。触发后，AI 会自动读取项目说明、同步进度、创建剧情拆解、补充角色档案，并生成最终内容。

这部分对 ClayMore 的启发是：稳定流程不要长期靠临场提示词，而要沉淀为工具或规则。

适合封装的任务包括：

- 从 Cubox 新资料生成 raw 副本、分类和整理日志。
- 从 B 站链接获取 metadata、转写稿和主题页草稿。
- 从一批 raw 资料汇总主题页。
- 对规则页做 link audit 和重复内容检查。

### 4. 用 Git 保护本地知识库

作者特别强调：当 AI 开始修改大量本地文件时，Git 是安全网。

基本做法：

- 初始化 Obsidian 仓库。
- 使用 Obsidian Git 插件或命令行定期提交 / 推送。
- 连接私有 GitHub 仓库。
- 在高风险整理或批量修改前先确认 Git 状态。

对 ClayMore 来说，这一点已经很实际：AI 会新增、移动、整合页面，必须通过 Git diff、日志和 lint 把改动变得可追踪。

### 5. 内容生产可以接入排版、配图和发布

视频后半段展示了把文章交给 AI 做公众号式排版、配图、预览，并进一步同步到微信公众平台或飞书的路径。

这部分不必直接照搬，但可以拆成三层：

- 文章层：结构、标题、分节、摘要、配图位置。
- 资产层：图片提示词、图片文件、链接和引用。
- 发布层：公众号预览、飞书同步、CLI 自动提交。

ClayMore 当前更需要先稳定“资料整理 -> 主题页沉淀”，发布自动化可以暂时放在后续阶段。

## 和 Karpathy Wiki 的差异

| 维度 | Karpathy Wiki 取向 | 本视频取向 | ClayMore 可采用方式 |
|---|---|---|---|
| 知识来源 | 网页、资料、链接收集 | 本地项目文件 + AI 可读取上下文 | raw 来源和主题项目包并存 |
| 核心动作 | 收集、归档、引用 | 理解、执行、反馈、迭代 | 每次整理都回写规则和主题页 |
| AI 角色 | 辅助检索与总结 | 持续执行项目工作流 | 让 AI 遵守 AGENTS 和规则入口 |
| 产物形态 | 静态 wiki 页面 | 动态创作/发布系统 | 先做可维护 wiki，再逐步自动化 |
| 风险控制 | 资料可追溯 | Git + 规则 + 反馈闭环 | Git diff、lint、整理日志 |

## 可迁移到 ClayMore 的改造

### 立即采用

- B 站资料不能只保留 Cubox 标题卡片，必须进入 `metadata -> 字幕/转写 -> 主题页`。
- 每条重要视频都要区分 `title-only`、`metadata`、`partial`、`full`。
- 专有名词不确定时，不写死到结论里，放到“待核验”。
- 主题页要写“可复用动作”，而不是只写视频摘要。

### 短期可做

- 为 Cubox 整理流程创建一个固定的“单条资料处理报告模板”。
- 为 B 站、抖音等视频建立统一字段：来源、metadata、转写文件、主题页、ASR 风险。
- 给长期主题建立“总入口 + 子页面”的结构，减少规则页和方法页之间的跳转成本。

### 暂不采用

- 自动发布到公众号 / 飞书。
- 自动配图和风格化排版。
- 复杂创作项目的多角色状态管理。

这些更适合在 ClayMore 的整理流程稳定后，再作为单独工具链实验。
